## 1. 미국 큰손 데이터 — 13F 기관 보유 + 내부자 거래 (고래추적 "미국" 탭)

- **목적**: 고래추적 메뉴의 개념("큰손이 어느 쪽에 서 있나")을 미국주식으로 완성. 1차분에서 한국 탭(외국인·기관 수급)만 만들고 미국 탭은 비워둠.
- **소스**: SEC EDGAR 공식 API (무료, User-Agent 헤더만 요구, 공시 원본 영구 보관 → 몇 년치 소급 가능)
  - 13F-HR: 기관 분기 보유 현황 (분기 지연 45일 — 실시간성 낮음에 유의)
  - Form 4: 내부자 매매 (거래 후 2영업일 내 공시 — 상대적으로 신선)
- **대략 구성**: EDGAR full-text/submissions API 수집기 → 주요 기관(예: 상위 운용사 화이트리스트) 분기 보유 변화 + 관심 종목 내부자 순매수 테이블 → 고래추적 "미국" 탭
- **착수 판단 기준**:
  - 1차분 한국 탭이 배포되고 실제로 쓰이는지 확인 후 (같은 메뉴의 수요 검증)
  - 13F의 45일 지연이 사용 목적에 맞는지 사용자 판단 필요 — "지금 큰손 방향"이 목적이면 Form 4 위주로 범위 축소 권장
- **예상 난점**: 13F XML 파싱(기관별 포맷 편차), 관심 기관/종목 목록 선정이 UX의 절반

## 2. KRX 프로그램매매

- **목적**: 한국 수급 데이터 보강 (외국인·기관 순매수의 하위 분해 — 차익/비차익 프로그램)
- **소스**: KRX 정보데이터시스템 또는 네이버 금융 (과거분 조회 가능 → 소급 OK)
- **대략 구성**: 1차분의 `kr_investor_flow` 수집 job에 컬럼 추가가 아니라 **별도 테이블**로 (1차분 스키마 안정성 유지) → 고래추적 한국 탭에 보조 차트
- **착수 판단 기준**: 1차분 수급 차트를 실제로 써본 뒤 "프로그램매매 분해가 더 필요한가"를 판단. 수급 총량으로 충분하면 하지 않는다.

## 3. CVD 주식판 스크리너 (거래량·다이버전스 스크리너)

- **목적**: CVD 스크리너 메뉴의 개념(수급 이상 징후 스크리닝)을 주식으로 확장
- **소스**: 자체 적재 불필요 — 야후(미국)·네이버(한국)의 가격·거래량 히스토리에서 계산 (시계 없음)
- **대략 구성**: 거래량 급증(20일 평균 대비 배수) + RSI 다이버전스 + 52주 신고/신저가 근접 스크리닝. CVD 자체는 주식 무료 데이터로 불가능하므로 "CVD와 같은 역할"이지 같은 지표가 아님.
- **착수 판단 기준**: 스크리닝 대상 유니버스(어떤 종목 몇 개를 돌릴지)가 정해져야 시작 가능. 전 종목 스캔은 무료 소스 rate limit과 충돌하므로 유니버스 선정(예: KOSPI200 + S&P500 구성종목, 또는 관심종목 슬롯)이 선행 결정.
- **우선순위 낮음**: 1차분 회의에서도 효용 대비 비용 커서 후순위로 분류했던 항목.

## 4. 리서치 자동 분석에 주식 데이터 편입

- **목적**: 리서치 메뉴의 카테고리별 자동 분석이 주식 데이터(수급·거래대금·주식F&G·Put/Call)도 다루게 확장
- **소스**: 자체 수집 없음 — 1차분과 조기 구축 트랙이 쌓는 테이블(`kr_investor_flow`, `kr_market_volume`, `stock_fear_greed`, `cboe_putcall`, `index_shadow_judgments`)을 읽는 쪽
- **착수 판단 기준 (정량)**: 원천 데이터 최소 4주 적재 후. 그 전에 만들면 분석할 표본이 없다.
  - 1차분 배포일 + 4주가 되는 시점에 검토
- **대략 구성**: 기존 리서치 분석 job의 카테고리에 "주식 수급", "주식 심리" 추가. 분석 로직은 기존 코인 카테고리 패턴 복제.

## 착수 순서 권장

데이터 적재 현황에 따라 자연스러운 순서:

1. **리서치 편입** (4번) — 1차분 + 4주 시점에 자동으로 착수 조건 충족. 가장 먼저 트리거될 가능성 높음.
2. **미국 큰손** (1번) — 한국 탭 수요 확인 후. Form 4 축소판부터.
3. **프로그램매매** (2번) — 수급 차트 사용 경험이 판단 근거.
4. **CVD 주식판** (3번) — 유니버스 결정이라는 선행 문제가 있어 마지막.

각 항목은 착수 시점에 이 백로그를 기반으로 ouroboros 스펙 → 상세 plan 문서 → 구현 순서를 따른다.
